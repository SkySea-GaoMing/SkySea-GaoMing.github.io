类加载与反射机制

这一部分可以参考我的博客：深入JVM。很多开源框架都是通过反射完成的

JVM和类

运行一个Java程序时会启动一个Java进程，程序的所有状态信息都处在这个进程
中，都是用JVM分配的内存区，当JVM进程结束时，进程中的状态信息都会丢失。
还可以这样理解，先将Java代码翻译成字节码文件，Java虚拟机加载的是字节
码文件，每次对字节码文件的修改在进程结束时都会丢失

类的加载

当程序主动使用某个类时，如果该类还未被加载到内存中，则系统会通过加载、
连接和初始化三个步骤对该类进行初始化。类的class文件读入内存后，会在
内存中创建一个java.lang.Class对象，类实际上也是一种对象，都是java.lang
.Class的实例

类的连接

连接负责把类的二进制数据复制到JRE中

  • 验证 验证阶段用于检验被加载的类是否有正确的内部结构，并和其他类协调一致
  • 准备 类准备阶段则负责为类的类变量分配内存，并设置默认初始值
  • 解析 将类的二进制数据中的符号引用替换成直接引用

类的初始化

对类变量进行初始化，声明时指定和静态初始化块都会变为类初始化语句，会按照
顺序执行初始化语句。初始化一个类的步骤如下

  1. 如果这个类还没有加载连接就先加载连接
  2. 如果该类的直接父类还没有初始化就先初始化直接父类
  3. 如果类中有初始化语句则依次执行

类初始化的时机

通过以下6种方式使用类或接口时就会进行初始化，注意使用final的时候不会导
致初始化，final修饰的在编译时就已经确定的变量不会初始化，类的loadClass
方法加载类时只是加载，也不会导致初始化

  1. 创建类的实例 为某个类创建实例的方式包括：使用new操作符来创建实例，
    通过反射来创建实例，通过反序列化的方式来创建实例
  2. 调用某个类的类方法（静态方法）
  3. 访问某个类或接口的类变量，或为该类变量赋值
  4. 使用反射方式来强制创建某个类或接口对应的java.lang.C1ass对象 例如代
    码C1ass.forName(“Person”) ，如果系统还未初始化Person类，则这行代码
    将会导致该Person类被初始化并返回Person类对应的java.lang.C1ass对象
  5. 初始化某个类的子类，当初始化某个类的子类时，该子类的所有父类都会被初始化
  6. 直接使用java.exe命令来运行某个主类，当运行某个主类时，程序会先初始化
    该主类

类加载器

类加载器负责将class文件加载到内存中并生成java.lang.Class对象

类加载机制

类加载器负责加载所有的类，一旦一个类被加载入JVM，同一个类就不会再次载
入。一个载入JVM的类都有一个唯一标识，在Java中用一个类的全限定名作为标
识，在JVM中一个类用全限定名和类加载器结合起来作为标识，比如有一个pg包
下的person类，分别通过k1和k2两个加载器加载，那么标识(pg,person,k1)
(pg,person,k2)是不同的，JVM启动时会形成由三个类加载器组成的初始类加
载器层次结构，类加载器的父子关系并不是继承关系，是指加载器实例之间的
关系

  1. Bootstrap ClassLoader 根类加载器 加载核心类
  2. Externsion ClassLoader 扩展类加载器
  3. System ClassLoader系统类加载器

JVM的类加载机制主要有如下三种

  1. 全盘负责 一个类加载器负责加载某个类时，这个类依赖和引用的其它类也会
    被加载，除非显示使用另一个类加载器
  2. 父类委托 先让父类视图加载Class，只有父类加载器无法加载时才会让子类
    加载
  3. 缓存机制 保证所有加载过的Class都会被缓存，当程序需要某个类时，
    类加载器会现在缓存区寻找该类，当缓存中不存在时才会读取该类的二进
    制数据并转化为Class对象存入缓存，这就是为什么修改Class后需要重
    启JVM

通过反射查看类信息

Java中许多对象在运行时都会生成两种类型：编译时类型和运行时类型

获取Class对象

每个类被加载后系统都会为该类生成一个Class对象，Java中通常有三种方式
获取该Class对象

  1. Class.forName(“类的全限定名”)
  2. 调用某个类的.class属性获取，这种方式的优点有两个
  • 编译阶段就可以检查需要访问的Class对象是否存在
  • 无需调用方法，性能更好
  1. 调用某个对象的getClass()方法

获取特定的构造方法

  1. getConstructor() 返回此Class对象对应的带指定参数的public构造器
  2. getConstructors() 返回所有public构造器
  3. getDeclaredConstructor() 返回带指定参数的构造器，与访问权限无关

使用反射操作对象

通过反射可以创建对象，首先获取对象的构造器，然后调用构造器的newInstance
方法创造实例

如果想调用私有构造器

获取Class对象后也可以通过getMethods()或getMethod()获取全部方法
或者指定方法。返回值是Method数组或对象，每个Method对象对应一个方
法，通过该对象的invoke调用相应的方法

第一个参数是类的实例，method也可以是私有方法

访问成员变量值

通过Class对象的getFields()或getField()方法可以获得该类所有成员
变量，Field提供两组方法读取或设置成员变量值

  • getXxx(obj) 获取成员变量的值，Xxx对应8种基本数据类型，如果是引用类型
    则不需要Xxx
  • setXxx(Object obj,Xxx value) 设置成员变量值，Xxx对应8种基本数据类型
    ，如果是引用类型则不需要Xxx

异常

部分图片参考 https://blog.csdn.net/qq_29229567/article/details/80773970
异常机制就是当程序出现错误的时候的处理机制。异常和错误的区别就是异常
是可以被处理的，错误一般是无法处理的

异常的分类

异常可以分为运行时异常和编译异常

  • 运行时异常 可以通过编译不能通过运行，如NullPointerException、
    IndexOutOfBoundsException
  • 编译异常 从语法角度必须进行相应的强制处理异常否则不能通过编译，如
    IOException、SQLException

异常处理

Java的异常处理机制可以让程序具有极好的容错性，当程序出现问题时，系统
会自动生成一个异常对象来通知程序，从而实现将业务逻辑代码和错误处理代
码分离

捕获异常

将业务逻辑代码放在try块中，将错误处理逻辑放在catch块中处理

如果在try块中出现异常系统会自动生成一个异常，这个异常对象会提交给Java
运行环境，这个过程就叫做throw，也就是抛出异常。Java环境接收到异常后会
寻找能够处理异常的catch块，如果找到合适的catch块就会交给catch块来处
理，这个过程就叫做catch，也就是捕获异常。如果Java环境找不到捕获异常
的catch块，那么运行环境就会终止

异常类的继承体系

每个catch块都是专门用于处理该异常类及其子类的异常实例，所以首先会判断
异常实例是否属于catch块后的异常或其子类。接下来看一下RuntimeException
的常见子类
catch块应该遵循先处理小异常后处理大异常的规则，子类异常的catch块应该
放在父类异常的catch块前面

多异常捕获

Java7以前每个catch块只能捕获一种类型的异常，Java7之后一个catch块可以
捕获多种异常

访问异常信息

如果程序想要获取异常对象的相关信息，可以通过catch块的异常形参获得，接下来
看一下Throwable源码，其中定义了所有异常对象公有的方法

finally

有些情况下系统打开了一些物理资源，比如数据库连接、网络连接和磁盘文件，
这些物理资源都必须显示回收，注意垃圾回收机制回收的是JVM内存中的空间。
接下来分析一下在哪里回收资源，如果在try中回收资源，那么程序运行到一
半可能产生异常，那么之后的代码就不会执行。如果在catch中回收资源，可
能程序不会产生异常那么catch块也就不会执行。所以异常处理机制提供了
finally块来回收资源，finally块一定会被执行

异常处理语法结构中try块是必需的，可以没有catch块或finally块，但两者
必须出现其一，就算没有catch块finally块也一定会执行。如果在finally块
中使用return或throw语句会导致try和catch中的return和throw语句失效

当try或catch块遇到return或throw时，首先会判断是否有finally块，如果
有finally块那么先执行finally块中的内容然后跳转到return或throw，如
果finally块中有return或throw，那么就不会再跳转到try或catch块

异常处理的嵌套

在try块catch块或finally块中包含完整的异常处理流程的情形称为异常处理
的嵌套，一般来说不要超过两层嵌套

自动关闭资源的try

当使用finally关闭资源时代码比较臃肿，必须手动关闭并且编写异常处理机
制。Java7增强了try块的功能，允许在try块后加括号，括号内可以声明初始
化多个资源，这些资源就是需要显示关闭的资源，try 块在语句结束后自动
关闭这些资源。注意这些资源必须实现AutoCloseable或Closeable接口，
实现这两个接口就必须实现close方法

Java9再次增强try语句，不要求在try后的圆括号声明并创建资源，只需要
自动关闭的资源有final修饰

throws

使用throws抛出异常的思路是：如果当前方法不知道如何处理这个异常，那么
该异常应该由上一级调用者处理，如果main方法也不知道如何处理异常那么也
可以使用throws声明抛出异常，将异常交给JVM处理。JVM对异常的处理方式就
是打印异常的跟踪栈信息，并中止程序运行。throws只能在方法签名中使用，
可以声明抛出多个异常类

方法重写时抛出异常的限制

使用throws抛出异常时有一个限制，子类方法抛出的异常类型应该是父类异常
抛出的方法类型的子类后相同，抛出的异常数不能更多

throw

当程序出现错误时会自动抛出异常，此外程序也可以自行抛出异常，自行抛出
异常用throw完成，每次只能抛出一个异常实例，注意运行时异常可以显示捕
获也可以不理会，Checked异常必须显示捕获

自定义异常类

自定义异常类需要继承Exception，如果希望自定义Runtime异常可以继承
RuntimeException
